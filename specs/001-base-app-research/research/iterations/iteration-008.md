## Deferred Questions Resolved

### Q2: UI Framework + Component Library for Data-Heavy Dashboard
**Decision: ShadCN UI + Tremor charts -- no additional admin framework needed.**

ShadCN UI ships with next-saas-stripe-starter and provides all foundational UI primitives (Button, Card, Table, Dialog, Tabs, etc.). For a data-heavy ops dashboard, ShadCN UI alone covers approximately 80% of needs. The gap is **charts and data visualization** for analytics (views, engagement, platform performance). Tremor (MIT, built on Recharts, designed for dashboards) provides the missing 20% -- area charts, bar charts, KPI cards, sparklines -- and is fully compatible with ShadCN UI + Tailwind CSS.

AdminJS and Refine are overkill: they are full admin panel frameworks designed for CRUD-heavy back-office apps. viral-ops needs a custom ops dashboard, not a generated admin panel. The dashboard must show pipeline status, content calendar, platform analytics, and upload queue -- none of which fit auto-generated CRUD patterns.

[INFERENCE: based on next-saas-stripe-starter component inventory (iter 2) and dashboard requirements from README.md pipeline stages]

### Q4: Database and ORM Trade-offs
**Decision: Prisma + PostgreSQL -- no concerns for viral-ops schema.**

Prisma + PostgreSQL is the dominant pattern across all evaluated SaaS boilerplates (BoxyHQ, next-saas-stripe-starter, Open SaaS, Saasfly). The multi-platform schema designed in iteration 6 (6 tables + 1 view) is well within PostgreSQL's capabilities. Key considerations:

- **JSONB columns** for platform-specific metadata work natively in PostgreSQL and Prisma supports them via `Json` type
- **UUID primary keys** are PostgreSQL-native via `gen_random_uuid()`
- **Prisma migrations** handle schema evolution; the viral-ops schema is relational with clear foreign key relationships
- **n8n uses SQLite** by default (separate database) -- no conflict with the dashboard's PostgreSQL
- **No performance concern**: The schema handles content metadata and analytics, not high-frequency writes. Even at 100 videos/day across 4 platforms, this is ~400 rows/day in `platform_publishes` -- trivial for PostgreSQL

One consideration: Prisma's lack of raw SQL escape hatches is sometimes cited, but Prisma's `$queryRaw` and `$executeRaw` cover any edge cases. The iteration 6 schema SQL is directly expressible in Prisma schema format.

[SOURCE: iteration-002.md (next-saas-stripe-starter uses Prisma+Neon PG), iteration-006.md (multi-platform schema design)]

### Q6: Admin/Dashboard Framework
**Decision: next-saas-stripe-starter's built-in admin panel is sufficient for Phase 1. Build custom dashboard pages on top.**

next-saas-stripe-starter includes an admin panel with user roles and management. For viral-ops Phase 1 (single user, local), this is more than enough for user/settings management. The operational dashboard (content pipeline, upload queue, analytics) should be built as custom Next.js pages using ShadCN UI components + Tremor charts, not generated by an admin framework.

Rationale: Admin frameworks (AdminJS, Refine) optimize for database CRUD views. viral-ops needs:
1. Pipeline status board (what is generating, what is queued, what failed)
2. Content calendar view (cross-platform scheduling)
3. Platform analytics dashboard (per-platform metrics)
4. Upload queue management (retry, cancel, reschedule)
5. Account management (platform OAuth tokens)

All of these are custom views, not auto-generated from schema. Building them as Next.js pages with ShadCN UI + Tremor gives full control over UX.

[INFERENCE: based on next-saas-stripe-starter admin panel assessment (iter 2) and dashboard requirements from pipeline architecture (iter 5)]

### Content Pipeline Flow (end-to-end)

```
User clicks "Generate" in Dashboard
  │
  ▼
Dashboard POST → n8n Webhook Trigger
  │
  ▼
n8n Workflow executes sequentially:
  │
  ├─ 1. POST :8000/api/llm/generate_script     { topic, language: "th", style }
  │      → Returns: script text + scene descriptions
  │
  ├─ 2. POST :8000/api/tts/synthesize           { text, voice: "th-TH-PremwadeeNeural" }
  │      → Returns: audio file path + duration
  │
  ├─ 3. POST :8000/api/image/generate           { prompts[], workflow: "flux-default" }
  │      → Returns: image file paths (via ComfyUI or RunningHub)
  │
  ├─ 4. POST :8000/api/video/compose            { audio, images[], captions: true }
  │      → Returns: final video file path
  │
  ├─ 5. INSERT content + platform_publishes     (via Dashboard API or direct PG)
  │
  ├─ 6. Upload per platform (parallel branches):
  │      ├─ YouTube: Data API v3 upload (built-in n8n node)
  │      ├─ TikTok: HTTP → TikTokAutoUploader
  │      ├─ Instagram: HTTP → Meta Graph API
  │      └─ Facebook: HTTP → Video API
  │
  ├─ 7. Post-upload affiliate (where automated):
  │      ├─ Instagram: POST /{media-id}/product_tags    ← AUTOMATED
  │      ├─ TikTok: Generate affiliate link via API     ← PARTIAL (link only)
  │      └─ YouTube/Facebook: skip (manual)
  │
  └─ 8. Respond to Webhook → Dashboard updates status
```

### Risk Assessment

**Risk 1: TikTokAutoUploader ban risk (MEDIUM)**
- Impact: TikTok uploads blocked, accounts flagged
- Likelihood: MEDIUM -- stealth measures (Phantomwright, fingerprint spoofing) are robust but TikTok actively combats automation
- Mitigation: (a) Apply for official TikTok API in parallel; (b) Rate-limit uploads (max 3/day/account); (c) Use multiple accounts; (d) Monitor for detection signals
- Fallback: Manual TikTok upload via mobile app while waiting for official API approval

**Risk 2: Pixelle-Video maturity (MEDIUM-LOW)**
- Impact: Bugs, missing features, breaking changes in early-stage OSS
- Likelihood: MEDIUM-LOW -- v0.1.15 with 4.1k stars indicates active community but still pre-1.0
- Mitigation: (a) Pin to specific version; (b) Fork and maintain local patches; (c) The FastAPI layer means we interact via stable HTTP endpoints; (d) MoneyPrinterTurbo as fallback (though stock-footage only)
- Fallback: Wrap MoneyPrinterTurbo's API if Pixelle-Video stalls

**Risk 3: Platform API changes / deprecation (LOW-MEDIUM)**
- Impact: Upload or shopping workflows break
- Likelihood: LOW-MEDIUM -- Meta and Google APIs are stable; TikTok is the most volatile
- Mitigation: (a) n8n workflows are easy to update when API endpoints change; (b) JSONB metadata columns absorb API response format changes; (c) Abstract platform-specific logic into n8n sub-workflows
- Fallback: Manual upload for affected platform while workflow is updated
